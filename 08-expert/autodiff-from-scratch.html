Build the autodiff engine yourself

The rite of passage. You've used this site's autodiff engine since the backprop page — now write it from a blank file, and PyTorch stops being magic forever.

Before you start Backpropagation, cold. Open your editor: this page is a guided build, and copying defeats it. Reference solution: assets/js/ml/autodiff.js — but only after each attempt.

The contract

One class, Value, wrapping one number, such that:

const w = new Value(-0.5), x = new Value(2), b = new Value(0.4);
const loss = w.mul(x).add(b).sigmoid().sub(1).pow(2);
loss.backward();
console.log(w.grad); // exactly dLoss/dw — no numerical nudging

Whatever expression the user builds, calling backward() on the result must fill every participant's .grad with its true nudge-effect on that result. That's the entire spec of an autodiff engine — the rest is deciding what bookkeeping makes it inevitable.

The build, in five steps

Step 1 — a Value remembers where it came from

Try first: a class holding data (the number), grad (starts 0), and — the load-bearing part — for values produced by an operation, which values produced it. Every computation quietly builds a family tree.

class Value {
  constructor(data, children = []) {
    this.data = data;
    this.grad = 0;
    this._children = children;
    this._backward = () => {};   // filled in by whoever creates me
  }
}

_backward is the design's one stroke of elegance: each operation will install a tiny closure that knows how to pass blame to its own parents. Nobody ever needs a global list of derivative rules — each Value carries its own.

Step 2 — add and mul (write these before peeking)

Each operation does three things: compute the result, record the parentage, install the blame-passer. The local derivatives you need: for addition, both parents inherit the child's gradient unchanged (d(a+b)/da = 1); for multiplication, each parent gets the child's gradient times the other parent's value (d(a·b)/da = b).

add(other) {
  const out = new Value(this.data + other.data, [this, other]);
  out._backward = () => {
    this.grad  += out.grad;
    other.grad += out.grad;
  };
  return out;
}

mul(other) {
  const out = new Value(this.data * other.data, [this, other]);
  out._backward = () => {
    this.grad  += other.data * out.grad;   // chain rule, one link
    other.grad += this.data  * out.grad;
  };
  return out;
}

The bug you must not write: those are +=, not =. If a value is used twice — a.mul(a), or a weight feeding two neurons — blame arrives along two paths and must accumulate. With =, the second path silently overwrites the first, gradients come out subtly wrong, training limps, and nothing crashes. This exact bug has consumed real engineering careers' worth of debugging hours.

Step 3 — the nonlinearities

Same pattern, one parent each. The local slopes: tanh′ = 1 − tanh², sigmoid′ = s(1−s), relu′ = 1 if positive else 0, and for pow(n): n·xⁿ⁻¹ — high-school calculus's greatest hits. Note the pleasant trick in the first two: the derivative is expressed in terms of the output you already computed, so nothing extra needs saving.

tanh() {
  const t = Math.tanh(this.data);
  const out = new Value(t, [this]);
  out._backward = () => { this.grad += (1 - t * t) * out.grad; };
  return out;
}
Step 4 — backward(): the only subtle part

Now the reverse sweep. Two requirements hide here. First, seed the output: this.grad = 1 (a thing's effect on itself). Second — the subtle one — order. A node's _backward must fire only after its grad is complete, i.e. after every node downstream of it has fired. The standard solution: a topological sort of the family tree, then walk it in reverse.

backward() {
  const topo = [], seen = new Set();
  (function build(v) {
    if (seen.has(v)) return;
    seen.add(v);
    for (const c of v._children) build(c);
    topo.push(v);                 // parents land before children
  })(this);
  for (const v of topo) v.grad = 0;   // fresh start each call
  this.grad = 1;
  for (let i = topo.length - 1; i >= 0; i--) topo[i]._backward();
}

Why not just recurse backwards from the output? Because of diamonds: if a value's blame arrives via two paths and you fire it after the first, it passes incomplete blame upstream. The topological order is precisely the guarantee that never happens. (The seen set handles the same diamond on the way down.)

Step 5 — the victory lap: train something with it

Prove the engine by using it as a trainer — this loop is logistic regression with zero hand-derived gradients:

let w = new Value(0), b = new Value(0);
for (let step = 0; step < 200; step++) {
  let loss = new Value(0);
  for (const p of points) {                       // {x, y} pairs
    const prob = w.mul(p.x).add(b).sigmoid();
    loss = loss.add(prob.sub(p.y).pow(2));
  }
  loss.backward();
  w = new Value(w.data - 0.1 * w.grad);           // (or reuse & zero grads)
  b = new Value(b.data - 0.1 * b.grad);
}

Sit with what just happened: you changed the model by editing one line (the expression), and training still works, because gradients are derived from the expression automatically. That decoupling — model as expression, gradients for free — is THE productivity unlock of the deep learning era. PyTorch adds tensors, GPU kernels, and forty thousand engineering hours; the soul you have just written.

Verify yours the way this site verified its own: pick a gnarly expression, compare every .grad against (f(x+h) − f(x−h)) / 2h with h = 1e-6. If they match to five decimals — and the a.mul(a) case gives 2a — your engine is real.
