Training vs. testing: the honesty rule

Never grade a student on the exact questions they practiced. The same courtesy applies to models — and forgetting it is the most common way ML projects fool their own creators.

Before you start Data & features, and it helps to have seen a model get graded — the linear regression demo if you're curious, though this page stands alone.

The idea

A student who memorized the answer sheet aces the practice exam. Do they know the subject? You find out with one move: new questions.

ML copies that move exactly. Before training, split the notebook:

  • The training set (typically ~70–80% of rows) — the model practices on these; its settings are tuned to fit them.
  • The test set — locked in a drawer. The model never sees these during training. At the end, they're the real exam.

The grade on the training set tells you how well the model fit what it saw. The grade on the test set estimates what actually matters: how it will do out in the world, on data nobody has labels for yet. When the two grades disagree badly, the model memorized instead of learned — and only the split lets you catch it. Doing well on new data has a name: generalization. It is the entire point.

The rules of the drawer

The test set only works if it stays untouched. Three practical rules:

  • Split first, look later. Decide the split before any fitting. Rows must not leak between piles — the same customer appearing in both counts as leaking.
  • Use the test set once. If you peek at test grades and go back to adjust the model, then again and again, you're slowly training on the exam. For repeated tune-and-check cycles, carve out a third pile — the validation set — and burn the true test set only at the very end.
  • The split must mimic reality. If the model will face next year's data, test on the most recent rows, not a random shuffle — predicting the past from the future is easy and meaningless.
